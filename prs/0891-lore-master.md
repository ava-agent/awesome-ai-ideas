## 1. 项目概述与痛点量化

### 1.1 项目愿景

LoreMaster 是一款面向桌面角色扮演游戏（TRPG）主持人（GM/DM）的AI辅助平台，旨在通过多模型协作大语言模型、检索增强生成（RAG）、知识图谱与多模态语音技术，为GM提供从剧本创作、规则管理、实时叙事到战斗平衡的全流程智能化支持。

### 1.2 目标用户画像

#### 画像A：全职/半职职业GM（Pro GM）

- **年龄**：25-35岁，有3年以上跑团经验
- **痛点**：
  - 每周平均花费 **8-15小时** 准备剧本，其中世界观设定和NPC对话设计占60%
  - 即兴叙事时频繁卡壳，平均每场3-5次叙事断裂（每次恢复需2-5分钟）
  - 管理长期战役时，**30%以上** 的世界观细节出现前后矛盾
- **期望**：减少50%以上准备工作时间，保持叙事连贯性，提升玩家沉浸感
- **付费意愿**：高（¥50-100/月），对效率工具有强需求

#### 画像B：新手GM / 尝试性主持人

- **年龄**：18-28岁，TRPG经验不足1年
- **痛点**：
  - 规则体系（D&D 5e/CoC 7e）掌握不熟练，**70%** 的新手GM在战斗中需要频繁翻阅规则书
  - 战斗平衡失控率高达 **55%**（怪物过强导致团灭，或过弱导致无聊）
  - 不确定自己的叙事质量，缺乏即时反馈
- **期望**：降低入行门槛，获得实时规则提示和战斗建议
- **付费意愿**：中等（¥20-50/月），对免费层有强依赖

#### 画像C：在线跑团社区组织者

- **年龄**：22-40岁，管理3个以上固定团
- **痛点**：
  - 多团并行时角色/剧情交叉记忆困难
  - 远程跑团时工具链碎片化（Discord + Roll20 + DnDBeyond + 笔记软件）
  - 玩家到场率不稳定，需要快速调整剧情走向
- **期望**：一站式平台整合，多团管理能力
- **付费意愿**：高（团队版¥99/月），需要多人协作功能

### 1.3 市场规模与机会

### 2.2 多模型协作设计（Model Collaboration Pattern）

LoreMaster 并非简单调用单一LLM，而是采用 **"专家模型分工协作"** 模式，每个模型承担其最擅长的子任务：

| 模型 | 角色 | 职责 | 选型理由 |
|------|------|------|----------|
| GPT-4o / GPT-4.1 | **叙事主模型** | 剧情生成、NPC对话、氛围描写 | 创意写作能力强，长上下文（128K）适合维持叙事连贯 |
| Claude 3.5 Sonnet | **规则推理引擎** | 判定规则、裁决争议、生成 rulings | 逻辑推理准确，指令遵循度高，适合精确规则执行 |
| GLM-4 / 智谱API | **中文优化模型** | 中文世界观生成、本土化内容 | 中文表达自然，文化语境理解好 |
| Llama 3 (本地) | **离线/隐私模式** | 离线战斗判定、本地NPC闲聊 | 无需网络，隐私数据不出设备 |
| DALL-E 3 / SDXL | **视觉生成** | 场景插图、NPC肖像、地图辅助 | 多模态增强沉浸感 |

**协作流程示例** — 玩家发起战斗行动时：

```
玩家行动 → [Claude: 规则判定] → 合法性检查 + DC计算
                                    ↓
                              [GPT-4o: 叙事描写] → 生成战斗描述
                                    ↓
                              [战斗平衡器] → 动态调整怪物HP/策略
                                    ↓
                              [知识图谱] → 校验NPC行为一致性
                                    ↓
                              [TTS] → 语音播报叙事
                                    ↓
                              返回玩家：描述文本 + 语音 + 状态更新
```

**延迟目标**：端到端响应 < 2秒（p95），其中LLM调用 < 1.5秒。

### 2.3 RAG 规则书知识库

#### 数据管线

```
规则书PDF/EPUB → 文档解析(Apache Tika + 自定义TRPG解析器)
                     ↓
              结构化分块(章节→小节→规则条目)
                     ↓
              双路索引构建:
              ├─ 语义向量索引 (text-embedding-3-large, 1536d)
              └─ 关键词倒排索引 (BM25, Elasticsearch)
                     ↓
              检索时: 混合检索 (向量0.7 + BM25 0.3) + RRF重排序
                     ↓

```

#### 支持规则体系（首批）

- **D&D 5e**：玩家手册、怪物图鉴、地下城主指南
- **CoC 7e**：克苏鲁的呼唤规则书、调查员手册、守密人指南
- **FATE Core/Accelerated**：FATE核心规则
- **PF2e**（Phase 2）：开拓者第二版

### 2.4 知识图谱与一致性守护

#### 图谱模型设计

```cypher
// 实体类型
(:World)          // 世界观
(:Location)       // 地点
(:Character)      // 角色/NPC/PC
(:Item)           // 物品/装备
(:Event)          // 事件/历史事件
(:Faction)        // 阵营/组织
(:Rule)           // 规则条目

// 关系类型
-[:LOCATED_IN]->          // NPC在某个地点
-[:KNOWS]->               // 角色间认识关系
-[:OWNS]->                // 拥有物品
-[:PARTICIPATED_IN]->     // 参与事件
-[:MEMBER_OF]->           // 属于阵营
-[:RELATED_RULE]->        // 关联规则条目
-[:TEMPORAL_BEFORE]->     // 时间线顺序
-[:CONFLICTS_WITH]->      // 矛盾标记（一致性校验产出）
```

#### 一致性校验流程

```
LLM生成内容 → 实体抽取(NER) → 图谱查询(已有实体/关系)
                                    ↓
                          ├─ 新实体 → 图谱写入 + 编号
                          ├─ 已有实体 → 属性/关系校验
                          │     ├─ 一致 → 放行
                          │     └─ 矛盾 → 触发修正建议 → GM确认
                          └─ 模糊匹配 → 消歧提示 → GM选择
```

## 3. 核心模块详细设计

### 3.1 模块一：智能剧本工坊（Scenario Workshop）

#### 功能描述

GM输入基础设定（风格、等级、时长、玩家数），AI生成完整剧本框架，包括：
- 主线剧情大纲（三幕式结构）
- 关键NPC档案（性格、动机、秘密、对话风格）
- 场景描述模板（环境、氛围、线索）
- 分支剧情节点树（玩家选择导致的剧情分歧）
- 战斗遭遇设计（怪物配置、地形、奖励）

#### 生成流程

```
┌─────────────┐     ┌──────────────┐     ┌──────────────────┐
│ GM输入设定   │────→│ 大纲生成器    │────→│ 分支剧情树生成器  │
│ · 风格偏好  │     │ (GPT-4o)     │     │ (GPT-4o + Tree)  │
│ · 玩家等级  │     │ · 三幕结构    │     │ · 节点生成       │
│ · 团队人数  │     │ · 冲突设计    │     │ · 条件分支       │
│ · 时长预期  │     │ · 反转点      │     │ · 后果链         │
└─────────────┘     └──────────────┘     └───────┬──────────┘
                                                 │
                    ┌──────────────┐     ┌───────▼──────────┐
                    │ 规则校验器    │←────│ NPC/遭遇生成器    │
                    │ (Claude)     │     │ (Claude + RAG)   │
                    │ · DC合理性   │     │ · 怪物属性查表   │
                    │ · 奖励平衡   │     │ · 战利品合规     │
                    │ · 规则兼容   │     │ · NPC对话样本    │
                    └──────────────┘     └──────────────────┘
```

#### 关键技术决策

- **大纲生成**：使用结构化JSON输出（function calling），确保格式可控
- **分支树**：限制最大深度为5层，每层最多3个分支，避免组合爆炸
- **迭代优化**：GM可对任意节点进行"重新生成"或"微调"，保留其他部分

### 3.2 模块二：实时叙事引擎（Live Narrative Engine）

#### 功能描述

跑团过程中，GM描述场景/事件后，LoreMaster实时辅助生成：
- 环境氛围描写（视觉、听觉、嗅觉等多感官描述）
- NPC对话（基于角色卡性格、当前情绪、历史交互）
- 玩家行动结果的叙述化描写
- 突发事件的即兴生成（玩家脱轨时）

#### 实时叙事流程

```
WebSocket连接建立
        │
        ▼
┌───────────────────────────────────────────┐
│            场景状态机 (Scene FSM)          │
│                                           │
│  EXPLORATION ──→ SOCIAL ──→ COMBAT       │
│       ↑              │           │         │
│       └──────────────┴───────────┘         │
│              (状态转换触发上下文切换)        │
└───────────────────┬───────────────────────┘
                    │
        ┌───────────▼────────────┐
        │   上下文窗口管理器      │
        │   · 近10轮对话         │
        │   · 当前场景描述        │
        │   · 活跃NPC列表        │
        │   · 玩家状态摘要        │
        │   · 剧本节点上下文      │
        │   (总计 ≤ 16K tokens)  │
        └───────────┬────────────┘
                    │
        ┌───────────▼────────────┐
        │   叙事生成管线          │
        │   1. 意图识别(输入分类) │
        │   2. 上下文增强(RAG)    │
        │   3. 多模型协作生成     │
        │   4. 一致性校验(图谱)   │
        │   5. 语音合成(TTS)     │
        └───────────┬────────────┘
                    │
        ┌───────────▼────────────┐
        │   流式输出(SSE)         │
        │   · 文本逐句推送        │
        │   · 语音段落推送        │
        │   · 状态更新推送        │
        └────────────────────────┘
```

#### 防止"AI感"的策略

- **温度控制**：叙事描写 temperature=0.9，规则判定 temperature=0.2
- **风格模仿**：GM可上传自己过往的叙事文本，微调生成风格
- **禁止词汇表**：过滤 "As an AI..."、"众所周知" 等破坏沉浸的表述
- **叙事多样性评分**：对生成文本计算n-gram重复率，>30%时触发重生成

### 3.3 模块三：动态战斗平衡器（Dynamic Combat Balancer）

#### 功能描述

实时监测战斗状态，动态调整难度，确保战斗体验既不无聊也不绝望：

- **战前预估**：根据玩家等级/配置预估遭遇难度（CR评估）
- **战中监测**：实时追踪伤害输出、资源消耗、战斗轮次
- **动态调整**：当战斗过难/过易时，提供GM调整建议或自动微调

#### 平衡算法

```
输入: party_level, party_size, monster_cr, current_round, 
      total_damage_dealt, total_damage_taken, resources_remaining

// 难度评分 (0-100)
difficulty_score = f(monster_cr, party_level, party_size, 
                     terrain_advantage, environmental_hazards)

// 战斗节奏评估
if round > expected_rounds * 1.5:
    // 战斗拖沓，建议加速
    suggest("reduce_monster_hp_by_20%")
    
if avg_damage_per_round > expected * 1.5:
    // 玩家碾压，动态加强
    suggest("monster_second_wind | reinforcement_wave")
    
if total_damage_taken > party_total_hp * 0.6:
    // 危险信号
    if round < expected_rounds * 0.4:
        // 早期团灭风险
        suggest("npc_intervention | escape_route_hint | 
                reduce_monster_accuracy")
    else:
        // 合理的高难度战斗
        pass

// 资源消耗监控
if spell_slots_remaining < 20% and round < midpoint:
    suggest("short_rest_opportunity | consumable_reward")
```

#### 平衡器输出形式

- **静默模式**：仅GM可见的面板提示（推荐）
- **半自动模式**：自动执行低影响调整（如微调怪物HP），重大调整需GM确认
- **全自动模式**（Beta）：完全自动平衡，GM仅监督

### 3.4 模块四：世界观一致性守护（World Consistency Guardian）

#### 功能描述

长期战役中自动维护世界观内逻辑一致性，防止GM/LLM出现"记忆错误"：

- **实体追踪**：记录每个NPC的名字、外貌、性格、位置、状态
- **时间线管理**：维护事件发生顺序，防止时间矛盾
- **地理一致性**：NPC位置与地理关系校验
- **规则一致性**：角色能力与规则体系合规性检查
- **矛盾检测与修正**：实时检测矛盾并建议修正方案

#### 矛盾检测示例

```
GM/LLM输出: "铁匠约翰从身后掏出一把火焰剑递给玩家"
     │
     ▼
实体抽取: Character(铁匠约翰), Item(火焰剑), Action(递给)
     │
     ▼
图谱查询:
  - 铁匠约翰: 职业=铁匠, 等级=3, 已有装备=[铁锤, 皮围裙]
  - 火焰剑: 所有权=古代龙巢, 价值=5000gp
  - 铁匠约翰 距离 火焰剑 所在地 = 3000英里
     │
     ▼
矛盾检测:
  ⚠️ 矛盾1: 铁匠(等级3)不太可能拥有一把价值5000gp的传说级武器
  ⚠️ 矛盾2: 铁匠约翰当前位于橡木镇，火焰剑在龙焰山脉
     │
     ▼
修正建议:
  1. 改为"一把精良的钢剑"(普通装备, 合理)
  2. 改为"他从架子上取下一把剑, 剑刃微微发红"(暗示魔法但未确认传说)
  3. 保留火焰剑, 但增加剧情说明(如: "这是他师父留给他的唯一遗物")
```

### 3.5 模块五：多语言NPC语音系统（Multilingual NPC Voice）

#### 功能描述

为NPC生成独特的语音角色，支持GM在跑团时通过语音与玩家互动：

- **声纹定制**：每个NPC可配置音色（性别、年龄、语速、音调、口音）
- **实时TTS**：叙事文本实时转语音，延迟 < 500ms
- **多语言支持**：中文、英文、日文、精灵语/矮人语等奇幻语言
- **情感表达**：根据NPC情绪状态动态调整语音情感（愤怒、悲伤、恐惧、喜悦）

#### 技术栈

```
文本输入 → 情感分析 → 风格标记注入 → TTS引擎 → 音频流
                │                         │
                ▼                         ▼
          情感标签:              引擎选项:
          {emotion: anger,      ┌─ Edge-TTS (免费)
           intensity: 0.8,      ├─ Azure Neural Voice (Pro)
           tone: threatening}   ├─ ElevenLabs (高级)
                                └─ Fish Speech (本地部署)
```

### 4.1 竞品矩阵

### 4.2 差异化竞争优势

1. **"GM副驾"定位**：不替代GM，而是增强GM能力，这与AI Dungeon的"AI即GM"形成根本差异
2. **多模型协作**：避免单一模型在创意和精确性上的取舍，每个模型做最擅长的事
3. **知识图谱一致性**：竞品中唯一内置世界观一致性守护的系统
4. **中文原生**：对中文TRPG社区（纯美苹果园等）有天然优势
5. **开放生态**：核心引擎开源，社区可开发规则体系插件

### 4.3 竞品弱点利用

| 竞品 | 弱点 | LoreMaster应对 |
|------|------|----------------|
| AI Dungeon | 无规则体系，叙事质量不稳定，中文差 | 规则RAG + 多模型协作 + 中文原生 |
| Roll20 | 无AI辅助，地图编辑器复杂，订阅价格争议 | AI全流程辅助 + 简洁界面 + 灵活定价 |
| Fantasy Grounds | UI老旧，学习曲线陡峭，价格高 | 现代UI + 新手引导 + 免费层 |
| Foundry VTT | 技术门槛高（自托管），AI依赖社区插件 | 开箱即用 + 内置AI + 云端托管 |

## 7. 风险评估

### 7.1 技术风险

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|----------|
| LLM幻觉导致规则判定错误 | 高 | 高 | RAG强制引用 + Claude精确规则模型 + GM最终确认机制 |
| 多模型协作延迟超标 | 中 | 高 | 异步并行调用 + 流式输出 + 本地模型备选 + 预生成缓存 |
| 知识图谱规模爆炸 | 中 | 中 | 实体生命周期管理 + 归档机制 + 分图存储（按战役隔离） |
| TTS延迟影响体验 | 中 | 中 | 预生成常用NPC语音 + 分段并行合成 + 降级为纯文本 |
| WebSocket大规模并发 | 低 | 高 | 连接池 + 消息队列(Kafka) + 水平扩展 |

### 7.2 业务风险

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|----------|
| 用户付费意愿低 | 中 | 高 | 免费层足够有价值(形成依赖) + 团队版差异化 + 内容市场补充 |
| LLM API成本失控 | 中 | 高 | 智能缓存 + 模型路由(简单任务用便宜模型) + 用量预警 |
| WotC/Chaosium版权风险 | 低 | 极高 | 仅索引规则摘要(合理使用) + 用户自行导入 + 法律咨询 |
| Roll20/FVTT推出AI功能 | 中 | 中 | 深耕GM体验(非VTT替代) + 争取被集成(插件/API) |
| 竞品快速复制 | 中 | 中 | 知识图谱壁垒 + 社区生态 + 先发用户积累 |

### 7.3 法律与合规风险

| 风险 | 缓解策略 |
|------|----------|
| 用户生成内容的版权归属 | 明确ToS：用户保留内容版权，平台获得非独占使用授权 |
| AI生成内容的可版权性 | 标注AI辅助生成 + 鼓励用户二次创作 |
| 数据隐私(GDPR/个人信息保护法) | 数据本地化 + 最小化采集 + 用户数据导出/删除权 |
| 未成年人保护 | 内容过滤 + 年龄验证 + 家长控制 |
